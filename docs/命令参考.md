# Scientex CLI Command Reference

This page contains the longer command catalog. The README keeps only quick-start and high-level guidance.

All commands support `-f json` for machine-readable output.

## Auth

```bash
scitex login
scitex status
scitex logout
```

`scitex login` 会输出飞书认证链接并在后台启动 token 轮询。用户完成浏览器授权后 token 自动保存，稍后运行 `scitex status` 检查结果。

```bash
scitex inventory item <ITEM_ID>
scitex inventory create-item item.json
scitex inventory update-item <ITEM_ID> item_update.json
scitex inventory disable-item <ITEM_ID>

scitex inventory create-stock stock.json
scitex inventory checkin <ID> --quantity 5 --purpose "restock" [--lab-id <LAB_ID>]
scitex inventory checkout <ID> --quantity 2 --purpose "PCR" --experiment-ref "EXP-001" --task-id <TASK_ID> --part-id <PART_ID> --requirement-key pcr.dntp [--lab-id <LAB_ID>]
scitex inventory checkout-item <ITEM_ID> --quantity 2 --purpose "PCR" --experiment-ref "EXP-001" --task-id <TASK_ID> --part-id <PART_ID> --requirement-key pcr.dntp
scitex inventory adjust <ID> --quantity -1 --type loss --reason "tube leaked"

scitex inventory check requirements.json
scitex inventory locations
scitex inventory create-location "Freezer A" [--parent-id <ID>] [--lab-id <LAB_ID>]
```

Inventory mutation `--quantity` values use the item's `usage_unit` amount, not the package `unit`, unless those two fields are the same.

## Admin

```bash
scitex admin task-types create task-type.json
scitex admin task-types delete <TASK_TYPE_ID>
scitex admin task-types staff list <TASK_TYPE_ID>
scitex admin task-types staff add <TASK_TYPE_ID> <USER_ID>
scitex admin task-types staff remove <TASK_TYPE_ID> <USER_ID>
```

- Admin commands manage platform task type catalog definitions.
- They do not create executable lab tasks; use `scitex tasks create` or `scitex tasks create-workflow` for task execution.
- Task type staff binding is catalog configuration; task instance assignment still uses workflow `assignee_ids`.
- If the backend rejects the request with a permission error, the CLI reports that the current account lacks permission for the admin operation.

### Task Creation Notes

- Use `scitex tasks create` for single-stage tasks with one root `task_type_id`.
- Use `scitex tasks create-workflow` for multi-stage tasks.
- Workflow payloads should use `parts[*].task_type_id` instead of a root `task_type_id`.
- Each workflow part must have a unique `client_key`.
- Dependencies should reference `client_key` values.

Workflow example:

```json
{
  "title": "Tm compute then QC review",
  "parts": [
    {
      "client_key": "tm_compute",
      "name": "Tm compute",
      "task_type_id": "compute-task-type-uuid",
      "input_data": {
        "sequence": "ATGGTCTCAGGAAACCTAGACCCAGAAAAACACGAATGG"
      },
      "sort_order": 10
    },
    {
      "client_key": "qc_review",
      "name": "QC review",
      "task_type_id": "staff-task-type-uuid",
      "input_data": {
        "sample_ids": ["S001", "S002"]
      },
      "assignee_ids": ["user-uuid-1"],
      "sort_order": 20
    }
  ],
  "dependencies": [
    {
      "prerequisite_client_key": "tm_compute",
      "dependent_client_key": "qc_review",
      "condition_type": "completed"
    }
  ]
}
```

### Task Inspection Notes

- Use `scitex tasks workflow <TASK_ID>` to inspect workflow structure, parts, dependencies, and assignments.
- `scitex tasks results <TASK_ID>` is workflow-aware:
  - single-stage compute tasks show compute output
  - single-stage staff tasks show submitted results
  - workflow tasks show results grouped by part, using `part.output_data` for compute stages and submitted results for staff stages
